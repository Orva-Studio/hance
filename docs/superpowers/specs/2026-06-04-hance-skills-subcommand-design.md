# Design: `hance skills` subcommand

**Date:** 2026-06-04
**Status:** Approved (pending spec review)

## Summary

Ship the Hance agent skill *inside* the CLI so any agent harness can pull
version-matched instructions at runtime via `hance skills`, modeled on
`agent-browser skills`. No install step, no symlink into `~/.claude/skills`, no
drift between the CLI release and the skill content it ships.

The skill content already exists at `skills/hance/`. This work flattens it to
the `skills/` root, bakes it into the build artifact via codegen, and exposes it
through a new `skills` subcommand.

## Goals

- `hance skills` serves the skill's router doc as the agent entry point.
- Content is identical whether Hance is run as the compiled single binary
  (GitHub releases) or the npm package (`@orva-studio/hance` via bunx/npx).
- One source of truth for skill content, consumed by both the CLI and the
  repo's Claude Code auto-load.
- Zero runtime filesystem dependency (the compiled binary has no `node_modules`
  beside it).

## Non-goals

- Multiple distinct top-level skills (only one skill today — YAGNI). A
  top-level namespace dir is reintroduced only if a genuinely separate skill is
  added later.
- Auto-registering into the user's `~/.claude/skills`.
- Bundling an LLM call inside the CLI — agents consume the markdown; the CLI
  only serves it.

## Source layout (single source of truth)

Move `skills/hance/*` up to the `skills/` root:

```
skills/
  SKILL.md            # router table + runner-selection rules ("start here")
  subcommands/        # run, try, refine, batch, setup, ui
  references/         # grading, preset-index, compare-page
  README.md           # updated: documents `hance skills`, drops match-look/symlink-only story
```

Names must be unique across `subcommands/` + `references/` (they are today); the
codegen enforces this.

## Embedding — codegen (`scripts/gen-skills.ts`)

A compiled binary cannot read `skills/*.md` off disk at runtime — there is no
filesystem next to it. The codegen turns the loose markdown into one importable
TS module that is baked into the JS bundle, covering both the binary and npm
distributions with one mechanism.

Steps:

1. Glob `skills/**/*.md`.
2. For each file derive:
   - `name` = filename without `.md`
   - `kind` = `root` (SKILL.md) | `subcommand` | `reference`
   - `description` = first heading/sentence (used by `list`)
3. Abort the build on a name collision across folders.
4. Emit `packages/cli/src/skills.generated.ts`:

```ts
// AUTO-GENERATED by scripts/gen-skills.ts — do not edit
export const SKILL_ROOT = "…full SKILL.md text…";
export const SKILLS: Record<string, { kind: string; description: string; content: string }> = {
  refine:  { kind: "subcommand", description: "Tune an existing grade", content: "…" },
  grading: { kind: "reference",  description: "Color grading reference", content: "…" },
  // …
};
```

`hance skills get refine` is then just `SKILLS["refine"].content`.

**Generated file is git-ignored** and built fresh; wired into `bun run build`
*before* the `--compile` step and into npm `prepublish`. A drift-guard test
(below) regenerates and compares so committed content can't silently lag.

## Command surface (`packages/cli/src/commands/skills.ts`)

Follows the existing `commands/preset.ts` pattern (arg parser + help text +
handler).

```
hance skills              → print SKILL_ROOT (the router; "start here")
hance skills list         → table of name + description, grouped subcommand/reference
hance skills get <name>   → that doc's markdown to stdout
hance skills path         → extract embedded skills to cache dir, print the path
```

- Add `"skills"` to the `Subcommand` union and `resolveSubcommand` in `cli.ts`.
- `hance --help` gains: `Start here (for AI agents): hance skills`.

## `path` behavior

Content is embedded, so `path` extracts on demand to a version-pinned cache dir
`~/.cache/hance/skills/<version>/` (honoring a `HANCE_SKILLS_DIR` override),
writing only if absent, then prints the dir. Gives agent-browser parity for
anyone wanting files on disk without writing on every invocation.

## Error handling

- `get <unknown>` → exit 1, message listing valid names.
- `skills` with no further args → print SKILL_ROOT (not an error).
- Defensive: empty manifest at runtime → clear error (should be impossible once
  codegen runs in build).

## Testing

- **Unit** (`packages/cli/__tests__/skills.test.ts`): `list` includes every
  generated entry; `get <known>` returns matching content; `get <unknown>`
  throws; arg parsing for `list`/`get`/`path`.
- **Drift guard**: regenerate the manifest in-test from `skills/**/*.md` and
  assert it matches what the CLI serves — adding a doc without rebuilding fails
  CI.
- **E2E** (`packages/cli/__tests__/e2e/`): `hance skills`, `hance skills get
  refine`, `hance skills path` against the built binary.

## Docs / package updates (in scope)

- Root `README.md` and `skills/README.md`: document `hance skills`; drop the
  `match-look` reference and the symlink-only install story.
- `cli.ts` `--help` `Start here` line.
- `package.json` `files` field: ensure `skills/` ships in the npm tarball;
  wire codegen into `build` and `prepublish`.

## Follow-up (out of scope)

- `https://hance.video/docs/agent/overview` agent guide should point at
  `hance skills` as the entry. Likely a separate repo — tracked as a follow-up.
