# Derive CLI flags & renderer defaults from EFFECT_SCHEMA

Issue: [#61](https://github.com/Orva-Studio/hance/issues/61) (incorporates #63)

## Problem

Effect parameter metadata (keys, ranges, defaults, choices) is duplicated across
several files instead of derived from the single `EFFECT_SCHEMA` source
(`packages/core/src/schema.ts`):

1. **Schema** — the canonical `{ default, min, max, choices, ... }`.
2. **CLI** (`packages/cli/src/effect-flags.ts`) — hand-written `EFFECT_HELP_TEXT`,
   `KNOWN_FLAGS`, `BOOLEAN_FLAGS`, and a per-flag `switch` re-listing every key,
   range, and default.
3. **Renderers** — inline fallbacks like `num("halation-amount", 0.25)` in
   `packages/ui/app/gpu/renderer.ts` and the Rust equivalent in
   `packages/wgpu/src/`.

Adding or removing one option (e.g. `halation-hue`, #48) is a 3–4 file change and
the copies drift silently. The real-world cost: the Rust export once defaulted
`halation-hue` to `0.5` (cyan) while everything else used `0.04` (warm), shipping
cyan halation when the param was omitted (#64).

Two further pieces from #63 are folded in:

- **Internal magic constants** (not exposed as params) are hardcoded separately in
  each renderer: the halation threshold band and the blur sigma factor.
- No **parity guard** catches the drift in CI.

## Goal / Acceptance

- Adding or removing an effect option is a **one-file change in the schema**; CLI
  flags/help/validation and renderer defaults follow automatically.
- **No behavior change** for existing flags/defaults — existing parse/snapshot and
  preset tests stay green.
- Shared internal constants live in one place consumed by both renderers; a test
  catches drift.

## Design

### Part 1 — Schema gains help metadata

Extend the `OptionDef` types in `packages/core/src/schema.ts`:

- `description: string` (required) — the human help blurb, e.g.
  `"Tint hue 0-1 (~red-orange)"`.
- `flag?: string` — optional override when the CLI flag name differs from
  `--<key>`. Used only by `input-lut-profile` → `--input-lut`.
- `alias?: { flag: string; value: string | boolean; help: string }` — sugar flags.
  Used only by `--vlog` → `{ flag: "--vlog", value: "vlog", help: "Sugar for --input-lut vlog (Panasonic V-Log)" }` on the `input-lut-profile` option.

The user-facing range hint (`<0-1>`, `<-2 to 2>`, `<1000-15000>`) is derived from
`min`/`max`, not stored. Choice hints (`<rec709|vlog>`) derive from `choices`.

`getDefaults()` is unchanged and remains the single defaults source.

### Part 2 — CLI generated from schema

In `packages/cli/src/effect-flags.ts`:

- A small hand-maintained `NON_EFFECT_FLAGS` table holds the flags that are **not**
  effect params: `--output`/`-o`, `--preset`, `--codec`, `--crf`,
  `--encode-preset`, `--blend`, `--export`, `--no-config`, `--help`/`-h`. Each
  entry carries its validator and help line.
- Everything else is generated by iterating `EFFECT_SCHEMA`:
  - **`KNOWN_FLAGS`** = non-effect flags + each option's flag (`flag ?? --<key>`) +
    each group's `--<enableKey>` toggle + alias flags.
  - **`BOOLEAN_FLAGS`** = `type: "boolean"` options + all `enableKey` toggles +
    alias flags + `--no-config`.
  - **Help text** = per group: the `label` as a header, then each option as
    `--flag <range|choices> Description (default: X)`, then the `--<enableKey>`
    line. Replaces `EFFECT_HELP_TEXT` entirely.
  - **Validation** = `range` → numeric `min`/`max` from schema; `select` →
    membership in `choices`; `boolean` → presence.

The big per-flag `switch` is replaced by a generated lookup keyed on flag → option
(with type-directed parsing/validation). Aliases map their `flag` to
`overrides[optionKey] = value`. Enable toggles map `--<enableKey>` to
`overrides[enableKey] = true`.

`ParsedEffectFlags` and `parseEffectFlags`'s external signature are unchanged.

Outcome: deletes `EFFECT_HELP_TEXT`, the hand-listed effect entries in
`KNOWN_FLAGS`/`BOOLEAN_FLAGS`, and the per-flag `switch` (~150 lines).

### Part 3 — Centralize defaults

- **`applyPreset()`** (`packages/core/src/presets.ts`): seed the merge with
  schema defaults first —
  `const merged = { ...getDefaults(), ...defaults, ...named, ...overrides }` —
  then drop the `?? <literal>` fallbacks (use `Number(merged[key])`,
  `Boolean(merged[key])`, `merged[key] as T`). The literal defaults now live only
  in the schema.
- **UI server** (`packages/ui/server.ts`): `/api/look` and `/api/look/info`
  return `{ ...getDefaults(), ...lookParams }` so the renderer receives
  fully-populated params.
- **UI App** (`packages/ui/app/App.tsx`): initialize `params` state from
  `getDefaults()` (merged under the disable-all / look state as appropriate) so
  the first render is also fully populated.
- **TS renderer** (`packages/ui/app/gpu/renderer.ts`): params now always
  populated, so the `num/bool` fallback literals are dead — drop them. Keep the
  `num`/`bool` helpers for type coercion with neutral fallbacks (`0` / `false`).

### Part 4 — Shared internal constants + parity guard (#63)

- New `packages/core/constants/render.json`:
  ```json
  { "halationThreshold": [0.65, 0.75], "blurSigmaFactor": 0.5 }
  ```
  These are the **current, already-consistent** values (the old `0.4/0.9` TS vs
  `0.65/0.75` Rust drift is already gone).
- **TS renderer** imports `render.json` and uses
  `halationThreshold` / `blurSigmaFactor` in place of the inline `0.65, 0.75` and
  `radius * 0.5`.
- **Rust renderer** loads the same file via `include_str!("../../core/constants/render.json")`
  + `serde_json` (the mechanism already used for the `.wgsl` shaders), replacing
  the inline `[0.65, 0.75]` and `radius * 0.5`.
- **Parity guard test** (bun): asserts `render.json` equals the expected literals,
  and asserts neither `renderer.ts` nor `renderer.rs` still contains the raw
  `0.65`/`0.75` threshold literals or `* 0.5` sigma for these passes — so editing
  one renderer without the shared file fails CI.

## Out of scope

- Making the **Rust renderer import the schema** — it keeps reading param keys
  directly. Centralizing defaults (Part 3) is what keeps it consistent.
- **Stripping Rust's `num(key, fallback)` literals.** With Part 3 the params
  always arrive populated, so they are dead and already consistent; removing them
  is a behavior-neutral cleanup with no drift benefit and is skipped to minimize
  risk. (Revisit if desired.)
- Any effect math / output changes.

## Testing

- **Safety net (must stay green):** existing CLI parse/snapshot tests, preset /
  `applyPreset` tests, e2e FFmpeg tests.
- **New tests:**
  - Generated help-text snapshot from the schema.
  - Generated `KNOWN_FLAGS` is a superset of the previously hardcoded flag set
    (regression guard).
  - `getDefaults()` matches the previously inlined `applyPreset` defaults for every
    key (equivalence guard).
  - `/api/look` response is fully populated with defaults.
  - Part 4 parity guard described above.
